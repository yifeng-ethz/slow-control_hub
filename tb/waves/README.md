# sc_hub Wave Reporting

This directory hosts the Phase-4 waveform reporting package for the standalone `sc_hub` verification harness.

## Artifact Tree

- `cases/<case-id>/meta.json`: machine-readable metadata for one published waveform case
- `cases/<case-id>/summary.md`: short human-readable summary for that case
- `generated/`: local VCD/UCDB/derived artifacts generated by the harness and ignored by git
- `gtkw/`: reusable GTKWave save-file stubs grouped by concern
- `templates/`: compatibility templates and generic save files
- `manifest.json`: generated aggregate index consumed by `index.html`
- `index.html`: static browser entry point for published wave cases

## Promoted Publication Set

The first cases worth publishing are:

- `SMOKE-BASIC`: directed bring-up reference
- `P001 -> T300`: promoted UVM performance anchor on Avalon-MM
- `CROSS-001 -> T356`: long mixed-feature Avalon case
- `CROSS-002 -> T357`: long mixed-feature AXI4 case
- one AXI4 baseline (`T124` or equivalent)
- one error-injection case from the `X` family

The goal is not to publish every VCD. The goal is to publish the cases that explain protocol, resource pressure, feature interaction, and failure signatures with the least duplication.

## Marker Convention

Major boundaries use `T0`, `T1`, `T2`, ... and sub-phase markers use semantic names such as `T5_FULL`, `T5_DROP`, `bp_stall_start`, or `ooo_event`.

Recommended baseline markers:

- `rst_deassert`
- `first_cmd`
- `first_reply`
- `last_reply`
- `drain_done`
- `bp_stall_start` / `bp_stall_end`
- `sva_*_fire` for assertion events

## Signal Group Conventions

The checked-in `.gtkw` stubs separate:

- packet link framing
- Avalon-MM bus activity
- AXI4 bus activity
- FIFO/backpressure levels
- overview bundles

Use decimal for counters and occupancy, hex for addresses and payload vectors, and add analog duplicates when queue depth or credit evolution matters.

## Minimal Workflow

1. Generate a VCD with `scripts/run_vcd_case.sh` or the underlying `make run_vcd` / `make run_uvm_vcd` targets.
2. Create or update a case package with `scripts/publish_wave_case.py`.
3. Generate `manifest.json` with `scripts/build_wave_manifest.py`.
4. Open `waves/index.html` in a browser to inspect the published package.

The current implementation provides the directory layout, metadata/index automation, a wrapper script, and reusable GTKWave save stubs. Per-case `.gtkw` curation is still required for high-polish publication.
